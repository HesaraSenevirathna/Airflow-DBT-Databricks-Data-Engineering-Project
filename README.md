# 🚀 End-to-End Data Engineering Pipeline: Ghost + Databricks + dbt + Airflow

## 📖 Project Overview
This project is an automated, end-to-end data engineering pipeline that integrates **Ghost (Agentic Postgres)**, **AWS S3**, **Databricks**, **dbt (Data Build Tool)**, and **Apache Airflow**. 

The pipeline extracts raw data, captures incremental changes (CDC), loads it into a Databricks data lakehouse, transforms it through a Medallion Architecture (Bronze, Silver, Gold), and tracks historical changes using Slowly Changing Dimensions (SCD Type 2). The entire workflow is orchestrated via Dockerized Apache Airflow.

---

## 🏗️ Core Concepts

### Change Data Capture (CDC) vs. Standard Incremental Load
**Standard Incremental Load (Batch):**
Runs a batch query (e.g., at midnight) using a watermark like `updated_at > '2026-07-09 23:59:59'`.
- *What it sees:* Finds new inserts (e.g., Order 102). 
- *The Problem:* If Order 101 was updated and then immediately deleted before the script ran, the batch script misses the deletion entirely. Order 101 sits in the warehouse indefinitely, causing data drift.

**Change Data Capture (CDC):**
CDC doesn't wait for midnight. It watches the database's native transaction logs in real-time.
- *What it sees:* A stream of events (10:00 -> INSERT Order 102; 11:00 -> UPDATE Order 101; 12:00 -> DELETE Order 101).
- *The Result:* The warehouse is perfectly accurate. It captures "upserts" (Update + Insert), ensuring idempotency. Deletions are processed accurately (e.g., via soft deletes `is_deleted = TRUE`).

## 🛠️ Step-by-Step Walkthrough

### Step 1: Database Provisioning with Ghost (Agentic Postgres)
Ghost is used to spin up a fully managed PostgreSQL database instantly.
1. **Installation:** Installed Ghost via `irm https://install.ghost.build/install.ps1 | iex` and Python `uv` (`pip install uv`).
2. **Environment Setup:** `uv init` -> `uv sync` -> `.venv/Scripts/activate`.
3. **Database Creation:** `ghost init`, then `ghost create --name ghost_db`. API keys and URLs are saved in a `.env` file (`ghost api-key create --name ghost_api --env > .env`).
4. **Schema Creation:** Used agentic prompts to create a `staging` schema and generated DDL scripts.
5. **Data Ingestion (`load_data.py`):** A custom Python script utilizing `psycopg2` and the `\copy` method to rapidly bulk-insert local CSV datasets into the Ghost database.
6. **Forking:** Instead of querying production directly, we run `ghost fork ghost_db ghost_db-fork` to create an isolated environment for testing.

### Step 2: AWS S3 & Databricks Auto Loader Setup
1. **S3 Connection:** Connected AWS S3 to Databricks by creating External Locations under Unity Catalog, passing secure IAM tokens.
2. **Data Ingestion:** Used Databricks' UI to configure ingestion. Selected the S3 bucket and used Databricks Auto Loader.
3. **How Updates Work:** To add new records, simply drop the old raw table, upload the updated CSV to the S3 bucket, and trigger the pipeline. Databricks auto-detects new data based on timestamps without running a full historical reload!
4. **SQL Warehouse:** Created a Serverless Starter Warehouse. The Server Hostname, HTTP Path, and generated Developer Access Token are used to connect dbt.

### Step 3: Data Transformation with dbt
Located in the `Airflow_DBT_Databricks_Data_Engineering_Project` directory. Setup via `uv add dbt-core dbt-databricks` and `dbt init`.

#### Important Configuration Files:
- **`profiles.yml`**: Tells dbt *how* to connect to Databricks. Contains the host, HTTP path, and access token. Usually located at `C:\Users\<User>\.dbt\profiles.yml`, but copied directly into the project IDE so Airflow can access it.
- **`dbt_project.yml`**: The brain of dbt. Defines the project name and dictates materialized configurations. Example:
  ```yaml
  models:
    airflow_dbt_databricks_data_engineering_project:
      bronze:
        +materialized: table
        +schema: bronze
      silver:
        +materialized: table
        +schema: silver
      gold:
        +materialized: table
        +schema: gold
        ephemeral:
          +materialized: ephemeral
  ```
- **`sources.yml`**: Maps dbt directly to the raw tables generated by Databricks in the `ghost` catalog.
- **`properties.yml`**: Contains data quality tests (e.g., checking for `not_null` or `unique` keys) to ensure data integrity during transformations.
- **`macros/`**: Contains reusable SQL functions. For example, `generate_schema_name.sql` is a custom macro that overrides dbt's default behavior (which normally prefixes schemas with your default target schema). By overriding it, we force dbt to use exact schema names like `bronze`, `silver`, and `gold` directly on Databricks.

#### The Medallion Architecture Layers:
1. **Bronze Layer (Raw & Cleansed):** Ingests raw data incrementally based on updated timestamps.
2. **Silver Layer (`obt.sql`):** The "One Big Table". This is a **Metadata-Driven Pipeline** using Jinja templating. Instead of writing massive join statements manually, a configuration dictionary loops through tables and automatically constructs the `LEFT JOIN` logic.
3. **Gold Layer (Ephemeral Models & Snapshots):**
   - **Ephemeral Models:** These models do *not* create physical tables or views in Databricks. dbt compiles their SQL and inlines them directly as Common Table Expressions (CTEs) into downstream models. 
   - **Snapshots (SCD Type 2):** Snapshots reference the ephemeral models (`relation: ref('eph_stores')`). They track historical changes using a `unique_key` and a `timestamp` strategy (checking `updated_at`). Current records have their end date set via `dbt_valid_to_current: "to_date('9999-12-31')"`.

*Note on adding new tables:* If a new table arrives in the source, you must manually create a new Bronze model, add its join logic to the Silver `obt.sql` configuration, and create Gold ephemeral/snapshot models.

### Step 4: Orchestration with Apache Airflow (Docker)
Airflow automates the execution of Databricks jobs and dbt models, tying the whole pipeline together.

#### Docker Setup & Installation:
- **Why Docker?** It provides a lightweight, isolated Virtual Machine to run the complex Airflow environment without messing up your local PC. Docker *must* be running continuously for the schedule to trigger automatically. (In real-life production, Airflow is hosted on cloud servers like AWS MWAA so it runs 24/7).
- **Getting `docker-compose.yaml`:** This file was downloaded directly from the [Official Airflow Documentation](https://airflow.apache.org/docs/apache-airflow/stable/howto/docker-compose/index.html) via curl. 
- **Modifications to `docker-compose.yaml`:** We customized the volumes block to map our local dbt project directory directly into the container (`- ${AIRFLOW_PROJ_DIR:-.}/Airflow_DBT_Databricks_Data_Engineering_Project:/opt/airflow/Airflow_DBT_Databricks_Data_Engineering_Project`).
- **`.env` File & Initialization:** Airflow requires an `.env` file to set permissions and secure secrets.
  - `AIRFLOW_UID`: Set to your local user ID (typically 50000 on Windows) so Docker has permission to write logs/dags.
  - `FERNET_KEY`: A cryptographic key generated via python (`cryptography.fernet.Fernet.generate_key()`) used by Airflow to encrypt database connection passwords. Read more at the [Fernet documentation](https://airflow.apache.org/docs/apache-airflow/stable/security/secrets/fernet.html).
- **`Dockerfile` & `requirements.txt`:** Instead of using the raw Airflow image, we use a custom `Dockerfile` to install dependencies (`dbt-databricks` and `databricks-sdk`) from `requirements.txt` into the container.
- **Running it:** Execute `docker compose up -d --build` in your terminal to build the custom image and spin up the Webserver, Scheduler, Triggerer, Redis, and Postgres backend.

#### The DAG (`orchestrate.py`):
This Python file defines the Directed Acyclic Graph (the pipeline workflow). The decorator `@dag(schedule="0 11 * * *", ...)` schedules the pipeline to run every day at 11:00 AM UTC.

Let's break down exactly what every task does in order of execution:
1. **`run_databricks_job`:** A python `@task` that uses the `databricks-sdk` to trigger an external Databricks Job (`run_now`). This Databricks job runs the Auto Loader pipeline, pulling the newest CSV files from S3 and updating the raw Databricks staging tables. The script uses `.result()` to wait until Databricks finishes ingesting the data before moving to the next step.
2. **`dbt_init`:** A `BashOperator` that runs `dbt clean` and `dbt deps`. This is critical because mounting local Windows folders into Linux Docker containers can cause stale dbt caches (`target/` folder), resulting in macro errors. This task wipes the cache clean and installs dbt packages.
3. **`source_cleansing`:** A `BashOperator` that runs `dbt run --select bronze`. This triggers the Bronze layer models to execute an incremental load, capturing only the newly arrived rows from the raw Databricks tables.
4. **`transform_silver`:** A `BashOperator` that runs `dbt run --select silver`. This dynamically merges the updated Bronze data into the metadata-driven "One Big Table" (`obt.sql`).
5. **`ingest_cdc`:** A `BashOperator` that runs `dbt snapshot`. Since our Silver layer was just updated with the freshest data, this task snapshots the Silver data to capture Slowly Changing Dimensions (SCD Type 2), archiving old records and starting new ones for any changes (e.g., a store name changing).
6. **`transform_gold`:** A `BashOperator` that runs `dbt run --select gold`. This compiles the ephemeral models and builds any final fact tables.
7. **`test_data`:** A `BashOperator` that runs `dbt test`. This runs the logic inside `properties.yml` against the Databricks tables to ensure data quality constraints (like no null primary keys) are strictly met.

**Final Execution Order:**
`run_databricks_job >> dbt_init >> source_cleansing >> transform_silver >> ingest_cdc >> transform_gold >> test_data`
